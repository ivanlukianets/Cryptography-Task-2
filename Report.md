## Теоретичнi вiдомостi щодо атаки часткового вгадування

### Про атаку
Атака часткового вгадування - це метод криптоаналізу потокових шифрів, у якому зловмисник вгадує значення невеликої 
підмножини змінних внутрішнього стану шифру, а потім відновлює решту стану, використовуючи функціональні 
зв'язки між змінними. Складність атаки визначається розміром вгадуваної підмножини: якщо вгадується k слів по n біт 
кожне, то складність становить 2^(k·n). Мета тут: знайти мінімальний такий базис, щоб складність атаки виявилась меншою 
за заявлену стійкість шифру.

### Інтстурент Autoguess
Autoguess це автоматизований інструмент для пошуку атак часткового вгадування на потокові шифри. Він приймає на вхід 
файл із функціональними зв'язками між змінними і за допомогою SAT-солвера знаходить мінімальний базис атаки.
Важливо згадати, що Autoguess трактує всі зв'язки як симетричні (оборотні). Тому якщо в реальній реалізації певна 
операція необоротна (наприклад, нелінійне перетворення T у Струмок), базис знайдений Autoguess може виявитись 
нереалізовним на практиці без заміни необоротного кроку на альтернативне рівняння.

### Приклади атак на інші шифри

* SNOW:
  * Це є найочевиднішим прикладом, адже SNOW є прототипом для Струмок. Перші guess-and-determine атаки на SNOW 
описали Hawkes і Rose у роботі "Guess-and-Determine Attacks on SNOW" 
Вони показали дві атаки: перша має складність 2^256 при обсязі даних 2^64 слів, друга - складність 2^224 при обсязі 
2^95 слів. Обидві менші за вичерпний перебір по ключу.
  * Пізніше Nemati Nia і Payandeh у роботі "The New Heuristic Guess and Determine Attack on SNOW 2.0" покращили цей 
  результат, використавши два допоміжних поліноми, їм вдалось зменшити розмір базису з 8 до 6 слів.

* Grain
  * Berbain, Gilbert і Maximov у роботі "Cryptanalysis of Grain" представили атаку на Grain v0, 
  яка потребує лише 2^43 операцій та 2^38 бітів гами для відновлення 80-бітного ключа. Метод комбінує лінійні 
  апроксимації функції зворотного зв'язку NFSR та фільтрувальної функції

### Про скрипт та файл який він згенерував
Для генерації файлу зв'язків написано скрипт relation_autoguess_generate.py. Файл strumok_11_ticks.txt містить 44 рядки.
```
# LFSR рекурсія (5 змінних - відводи +15, +13, +4):
S_{t+16}, S_{t+15}, S_{t+13}, S_{t+4}, S_t

# Keystream (4 змінні, Z_t - відомий):
Z_t, S_{t+15}, R1_t, R2_t

# Оновлення R1 (3 змінні):
R1_{t+1}, R2_t, S_{t+15}

# Оновлення R2/нелінійне T (2 змінні):
R2_{t+1}, R1_t

known
Z_0, Z_1, ... Z_10  <- відомі атакуючому без вгадування
```
### Результат Autoguess
Команда для запуску
```
autoguess --inputfile strumok_11_ticks.txt --solver sat --maxguess 7 --maxsteps 23
```

### Вгадані 7 слів базису
{S2, S6, S10, S11, S12, S13, S14}
* Примітка щодо вибору базису:
у Autoguess вважається, що всі математичні зв’язки в шифрі симетричні. Проте на практиці перетворення T є нелінійним, 
тому його важко «розвернути» назад. Щоб програма працювала, я обрала такий базис, який дозволяє відновити стан шифру 
через прості й оборотні операції
